# hira.page 文体ガイド

`src/blog/ja/` 配下の既存記事から抽出した、ブログ筆者の文体特徴をまとめたもの。`article-polish` SKILL から参照される。

清書するときは、このガイドの内容を**全部頭に入れた上で**、サンプル記事 (本ファイル末尾に列挙) を1〜2本ざっと読み返してリズムを掴んでから書き起こすこと。

---

## 基本トーン

- **ですます調** を基本とする。技術記事もエッセイも統一。
- 軽い断定 (「〜のはず」「〜だろう」「〜かも」) を時折混ぜて、文末のリズムを単調にしない。
- **過度に丁寧にしない**。「〜させていただきます」「〜という形になります」は使わない。
- 個人ブログとしての**主観・感想・好み**を隠さず書く。「個人的には」「正直」「めっちゃ好きです」のような一人称の感情表現は積極的に。

## 一人称

- 「私」「わたし」「ぼく」が**混在**。文脈で自然に切り替える。
- 技術記事では「私」、エッセイ的・感想多めの記事では「わたし」「ぼく」が出やすい。
- 元の下書きに一人称があれば、それを尊重する。

## 語尾と語感

よく使う語尾の例 (どれかに偏らず散らす):

- 「〜です」「〜ます」(基本)
- 「〜ですね」「〜ますね」(共感・余韻)
- 「〜と思います」「〜と感じています」(主観の明示)
- 「〜してます」「〜になってます」(口語的短縮)
- 「〜かもです」「〜かもしれません」(やわらかい推測)
- 「〜なんですよね」「〜なんですけど」(打ち明け感)

避ける語尾:

- 「〜となります」「〜となっております」(硬すぎる)
- 「〜です。〜です。〜です。」の3連続 (リズムが死ぬ)
- 体言止めの連発

## 口語表現の許容

砕けすぎない範囲で口語を混ぜる。書き言葉だけで固めない。

- 「めっちゃ」「ちゃんと」「ぜんぶ」「ぐらい」「やっぱり」「いっしょに」「いまさら」「ざっくり」「あえて」「ちなみに」「あいかわらず」「ありがたい」
- 「〜っちゃ〜」(例:「当然っちゃ当然」)
- カッコ書きの軽い補足 (例:「(3つくらい買いました)」「(当然っちゃ当然ですが)」)

## 強調

- `**太字**` を**要所で**使う。文中の最重要キーワードや結論部分。
- 固有名詞 (「**Dynamic Color**」など) や主張の核 (「**そのまま丼として使えて、さらに食洗機で洗える**」など) に。
- 1段落に太字を2つ以上入れない。打ちすぎると効かなくなる。
- `*斜体*` はほぼ使わない。
- `~~取り消し線~~` も極まれ。

## 絵文字・記号

- **絵文字はほぼ使わない**。本文・見出しに絵文字を散らさない。
- 例外: ごく稀に箇条書きの先頭で `★` `→` を使う程度。
- 顔文字も使わない。

## 段落の作り方

- 1段落1トピック。**2〜4文** が標準。
- 1文が長くなったら接続詞で切る。
- 段落間は必ず空行。
- 段落の長さがバラバラでもよい (リズム重視)。

## 見出し

- **h1 (`#`) は使わない**。frontmatter の `title` が自動でh1になる。
- `##` (h2) = 章、`###` (h3) = 節。`####` (h4) はあまり使わない。
- 見出しに記事のキーワードを自然に含める (SEO)。
- 見出しの語感は**結論っぽく** or **問いかけ調**:
  - 「私が Tailwind (と React) が嫌いな理由」(主張)
  - 「多言語対応のサイトを作るときにおすすめのフォント」(検索意図)
  - 「ガジェット系」「コンテンツ系」(セクション分け)
  - 「まとめと個人的な感想」(締め)
- 見出しに「!」「?」を入れてもよい (連発しない)。

## リスト

- 基本は `-` を使う (既存記事の傾向)。
- 商品リスト的なものでは `*` も混在 OK (Markdownでは同等)。
- 数字リストは手順説明や順序が意味を持つときだけ。
- ネストは2階層までが読みやすい。

## リンク・引用・出典

### インラインリンク

- `[テキスト](URL)` を多用。
- リンクテキストには「ここをクリック」のような無意味な語を使わず、リンク先の内容が分かる言葉を入れる (SEO・アクセシビリティ)。
- 1段落に2つ以上リンクを入れる場合は、リンクの目的を明確に書き分ける。

### 外部記事の参照

外部記事を本格的に紹介・参照する場合は `<EmbedCard>` を使う:

```mdx
<EmbedCard
    url="https://example.com/article"
    img="https://example.com/ogp.png"
    title="記事タイトル"
    site="example.com" />
```

- 直前または直後の段落でその記事の文脈・要点・自分の意見を書く流れ
- カードを連続で並べるときは間に1〜2文の説明を挟む

### 引用ブロック

外部の主張・データを引用する場合は `> ` ブロック + 出典リンク:

```mdx
> 引用文をここに。短く要点だけ。

[出典: 記事タイトル](https://example.com/article)
```

- 引用は**短く・要点だけ**。著作権配慮として記事本文の長い段落をまるごと引用しない。
- 必ず出典リンクを添える。

### 脚注

補足的なリファレンスは脚注で:

```mdx
本文に脚注をつける[^1]

[^1]: 脚注の説明と出典URL
```

### 参考リストの定型パターン

記事末尾や章末に「参考」のリストをまとめる場合:

```markdown
- [記事タイトル - サイト名](https://example.com/article)
- [別の参考記事 - サイト名](https://example.com/another)
```

または章の冒頭で:

```markdown
- [Goodbye SASS 👋, welcome back native CSS - DEV Community](https://dev.to/...)
- [最近Sassを使ってないなぁって話 | Webクリエイターボックス](https://www.webcreatorbox.com/...)
```

(`tailwind-react-styling-issues` の章冒頭が良い例)

## MDX コンポーネント慣習

import 文は frontmatter 直後に置く。空行を1つ挟む。

```mdx
import EmbedCard from '@/components/Blog/EmbedCard.astro';
```

### `<EmbedCard>` — 外部リンク・商品リンクのカード表示

```mdx
<EmbedCard
    url="https://example.com/"
    img="https://example.com/image.png"
    title="ページタイトル"
    site="example.com" />
```

- 商品レビュー記事ではほぼ毎節で使う。
- 直前または直後の段落でその商品 / 記事の説明を書く流れ。
- 1個のリンクを「インライン `[]()` リンク」と「`<EmbedCard>`」両方で書くことが多い (見出しでテキストリンク → 直後にカード)。

### 画像

- 相対パス基本: `![alt](./filename.png)` (記事フォルダ内に置く)
- alt はかならず書く (空文字でもよいが基本入れる)。検索エンジン・アクセシビリティ両方に効く
- 拡張子は内容に応じて選ぶ:
  - 静止画 → `.png` (UI スクショ・ロゴ) / `.jpg` (写真)
  - アニメーション → `.webp` (推奨。`.gif` も可)
  - 動画 → `public/video/` に置いて `<video src="/video/...">` (記事フォルダには動画を置かない方針)
- 外部画像 (Amazon・公式サイトなどから直接) を使う場合は絶対 URL でもよい

#### 画像の出典・キャプション

直後に `<small>` を置くのが慣例:

```mdx
![dynamic-color](./dynamic-color.gif)
<small>https://m3.material.io/foundations/customization</small>
```

参照リンク形式:

```mdx
<small class="reference">
    参照: <a href="https://unsplash.com/" target="_blank">Unsplash</a>
</small>
```

#### 仮画像パス (プレースホルダ) の慣習

清書時、まだ用意していない画像については **仮のパスを記事に入れておき、画像が必要なことを明示** する。あとから筆者が実画像を置けば差し替え不要で動くようにする。

ルール:

- **ファイル名は kebab-case で内容を示す**: `./dynamic-color.gif`, `./switchbot-air-purifier.jpg`, `./top-app-bars.png` のように、何の画像かが見て分かる名前
- **alt テキストには内容説明を必ず書く**: 「画像」「image」のような無意味な alt は NG
- **どんな画像が必要かを HTML コメントで添える** (筆者があとで用意するときに見られる):

```mdx
{/* TODO: ダッシュボードのスクショ。比較表が見える状態で */}
![ダッシュボード比較画面](./dashboard-comparison.png)
```

- MDX のコメントは `{/* ... */}` 形式 (HTMLコメント `<!-- ... -->` も動くが MDX 推奨は前者)
- 既に置かれている画像 (`cover.png` `thumbnail.png` 等) は触らない・パスを変えない
- 動画プレースホルダの場合は `public/video/{slug}-{name}.mp4` のような将来パスを書き、TODO コメントで何の動画かを明示

仮画像の出典 / キャプションは、実画像を入れるまでは「公式サイトより」程度の予定を書くか、TODO に入れて空欄でもよい。

### 動画・アニメーション

- 短いインタラクション説明 → アニメ webp/gif を記事フォルダに置く
- 長い動画・音声付き → `public/video/{name}.mp4` を作り `<video src="/video/{name}.mp4" controls />`
- 自動再生したい場合は `autoplay muted loop` (音は基本ミュート)

### 動画

- 同フォルダに置けないので、Webp / Gif アニメで代替するのが基本。
- どうしても動画なら `public/video/` に置いて `<video src="/video/...">`。

### コードブロック

- 言語指定を必ず付ける (`` ```js ``, `` ```ts ``, `` ```mdx ``, `` ```html `` など)。
- Shiki transformer 記法を必要に応じて使用 (`[!code ++]` `[!code highlight]` 等)。
- 設定済み機能: diff / highlight / word-highlight / focus / error-level。

### その他のHTML直書き

- `<small>` (出典・補足)
- `<details><summary>` (折りたたみ)
- `<b>` (軽い強調)
- `<dl><dt><dd>` (定義リスト)
- `<a class="download" href="..." download="...">` (ダウンロードリンク)
- Twitter埋め込み (`<blockquote class="twitter-tweet">` + script タグ)
- YouTube埋め込み (`<iframe>`)

## 接続表現の例

連続使用を避けつつ、以下を散らす:

- 順接: 「なので」「ですし」「そして」「また」
- 逆接: 「ただし」「とはいえ」「ですが」「一方で」「もちろん〜が」
- 強調: 「特に」「とくに」「なにより」
- 補足: 「ちなみに」「ところで」「ついでに」「あえて言うなら」
- 結論: 「つまり」「ようするに」「結局」

## 読者への語りかけ

たまに挟むと親しみが出る (連発しない):

- 「〜ですよね」「〜と思いませんか」
- 「〜してるひとも多いと思います」「〜のひとも多いはず」
- 「もし〜なら、〜してみてください」

## 記事の構成パターン

### 導入の典型

- 個人的なきっかけ・背景から入る ("毎年書いてる買ってよかった商品リスト、4年目です")
- 主題の前置き ("こちらが先日公開された 〜 の公式ページです")
- 主張の予告 ("〜について書きます" "〜に私が感じている課題点について解説します")
- いきなり結論をぶつけて読者を引き込むパターンもあり

### 本論

- h2 で章を立てる
- 必要に応じて h3 で節
- 各章は **(主張) → (具体例 or 引用 or コード) → (補足 or 反論への先回り)** のリズムが基本
- 画像・コード・カードで視覚的なリズムを作る

### 締めの典型

- 「まとめ」「まとめと個人的な感想」の h2 を立てて1〜2段落で振り返る
- 「まとめ」を立てず、最後の章で自然にフェードアウトするパターンも多い
- 締めには主観・感想・今後の期待を入れることが多い

## SEO 観点で意識すること

- title は検索意図のあるキーワードを含める (frontmatter は触らないが、本文の見出しに反映する意識)
- リード文 (冒頭1段落) に記事の主題キーワードを含める
- h2 / h3 に検索キーワードを自然に織り込む
- 内部リンク (`/blog/...`) と外部リンクをバランスよく
- 画像の alt を必ず書く
- 段落の長さを均し、読み流せるリズムにする

## 避けるべき表現 (まとめ)

- 「〜させていただきます」の連発 → 「〜します」
- 「〜することができます」 → 「〜できます」
- 「〜となります」 → 「〜になります」 or 「〜です」
- 「以下に〜を示します」 → 直接書く / 「こんな感じです:」
- 「皆さん」 → 「みなさん」 or 「ひとも多いと思います」
- 体言止めの連発
- AI翻訳調 (受け身の連続、無生物主語の主導)
- 過剰な絵文字・記号
